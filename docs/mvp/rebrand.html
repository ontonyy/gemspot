Status codes are part of the contractNest annotates non-default codes (@HttpCode(200) on login/refresh/oauth/logout/saved POSTs; 202 on events). Spring returns 200 for POST by default, so the only one needing an explicit @ResponseStatus(HttpStatus.ACCEPTED) is POST /events. Register returns 201 in Nest (default POST) — match it with @ResponseStatus(HttpStatus.CREATED) on register.

No global path prefix

Nest mounts controllers at the root (no /api prefix); the frontend calls ${VITE_API_URL}/places etc. Keep Spring at the root too — do not set server.servlet.context-path. Health stays at /health (override Actuator or use a plain controller) because Cloud Run's health check and the frontend both expect that exact path returning {"status":"ok"}.

Three Hibernate gotchas 1. String arrays — tags String[] maps to Postgres text[]. Use @JdbcTypeCode(SqlTypes.ARRAY) @Column(columnDefinition="text[]") List<String> tags (Hibernate 6).
2. JSON — Event.props is jsonb: @JdbcTypeCode(SqlTypes.JSON) Map<String,Object> props.
3. Application-assigned IDs — Place and Category use meaningful string ids, not generated. Annotate @Id with no @GeneratedValue; on save() of a new row Hibernate will INSERT (use persist semantics / implement Persistable if you hit select-before-insert overhead). UUID-keyed entities use @GeneratedValue(strategy=UUID) or assign in code as today.

Liquibase

An exact baseline already exists: Prisma generated backend/prisma/migrations/0001_init/migration.sql (all enums + tables). Port it rather than reverse-engineering from entities.

  • Master changelog src/main/resources/db/changelog/db.changelog-master.xml includes per-change files.
  • 0001-init.xml — translate the Prisma init SQL: create the four enum types (UserRole, PlaceStatus, SubmissionStatus, ReportStatus, ReportReason), all tables with their @@map names, FKs with the documented onDelete rules, and unique constraints / indexes.
  • Set ddl-auto: validate in JPA — Hibernate validates against the Liquibase-built schema and never mutates it. Liquibase owns DDL.
EnumsMap Postgres enum types to JPA with @Enumerated(EnumType.STRING) backed by columnDefinition, or define the columns as varchar with a check constraint if native PG enums prove fiddly with Hibernate. Either way the values on the wire stay the uppercased strings (ACTIVE, PENDING, …) the admin DTOs expect.

Auth — Spring Security + JWT

~1.5 days

A faithful port of the stateless dual-token scheme. No sessions, no server-side token store; logout is client-side. Two separate secrets, two TTLs.

Nest mechanismSpring equivalent
@nestjs/jwt signAsyncJJWT 0.12 — JwtService.issue()
bcryptjs hash/compare (rounds 10)BCryptPasswordEncoder(10)
JwtAuthGuard (Bearer verify)OncePerRequestFilter → JwtAuthFilter
@CurrentUser() decoratorSecurityContext principal / @AuthenticationPrincipal
AdminGuard (role check)@PreAuthorize("hasRole('ADMIN')") / authorizeHttpRequests
UnauthorizedException → 401AuthenticationEntryPoint → 401

Token claims — keep identical

  • Access: { sub:userId, email, role }, signed with JWT_SECRET, TTL JWT_ACCESS_TTL (15m).
  • Refresh: { sub:userId, typ:"refresh" }, signed with JWT_REFRESH_SECRET, TTL JWT_REFRESH_TTL (30d). Refresh verifies typ==="refresh" and rotates both tokens.
  • AuthResponseDto = { user:{id,email,name,role}, accessToken, refreshToken } — the shape authStore.ts persists. Do not change field names or casing.

SecurityConfig route rules

// stateless, CSRF off (JWT bearer, no cookies)
permitAll: /health, /places/**, /categories, /guides/**,
           /auth/** (except /auth/me), POST /events
authenticated: /auth/me, /saved/**, /submissions/**,
               /reports/**, /uploads
hasRole(ADMIN): /admin/**
Role mappingSpring's hasRole('ADMIN') expects an authority named ROLE_ADMIN. When building the Authentication in the filter, grant new SimpleGrantedAuthority("ROLE_" + claims.role). The role value in the JWT and DTOs stays the bare "ADMIN"/"CLIENT" the frontend reads.

Google OAuth

Port oauthGoogle: verify the ID token via GET https://oauth2.googleapis.com/tokeninfo?id_token= using Spring's RestClient (no extra dep), check aud == GOOGLE_CLIENT_ID and email_verified, then link-by-email (existing account gains the provider link, new email creates an OAuth-only account with null password). If GOOGLE_CLIENT_ID is unset, return 401 "not configured" — same as today.

Supabase Storage adapter

~1 day

Keep the storage seam: a StorageService interface (StoredFile save(file)) the UploadsController codes against, with a SupabaseStorageService implementation replacing local disk. Controller validation is unchanged: field name file, max 5 MB, mime in {jpeg,png,webp,gif}, returns { url }.

  • Create a public bucket (e.g. place-photos) in Supabase Storage.
  • Supabase exposes an S3-compatible endpoint — use the AWS SDK v2 S3Client pointed at https://<project>.supabase.co/storage/v1/s3, region eu-north-1, path-style access, with the Supabase S3 access key/secret. Upload object {uuid}.{ext}.
  • Return the public URL: https://<project>.supabase.co/storage/v1/object/public/place-photos/{name}. This becomes the url stored on submission/place photo rows.
  • Delete the old useStaticAssets('/uploads') path and ASSET_BASE_URL — photos are no longer served by the API.
Why this finally fixes photosRender's free disk was ephemeral, so uploads vanished on redeploy. Supabase Storage is durable object storage — uploaded photos persist across Cloud Run revisions and cold starts.

Containerize for Cloud Run

~½ day

Multi-stage Dockerfile: Gradle build stage → slim JRE 25 runtime. Cloud Run injects $PORT; the app must bind it.

# backend/Dockerfile
FROM gradle:9-jdk25 AS build
WORKDIR /app
COPY . .
RUN gradle bootJar --no-daemon

FROM eclipse-temurin:25-jre
WORKDIR /app
COPY --from=build /app/build/libs/*.jar app.jar
# Cloud Run sets $PORT (default 8080)
ENV PORT=8080
EXPOSE 8080
ENTRYPOINT ["java","-jar","app.jar"]

In application.yml: server.port: ${PORT:8080}. Expose health at /health (Cloud Run --startup-probe / liveness uses it). Keep the image lean so cold starts stay tolerable.

Cold start realityScale-to-zero means the first request after idle pays JVM + Spring startup (a few seconds). The SPA's withMockFallback masks this on read paths. If it bothers users, set Cloud Run min-instances=1 later (small always-on cost) — not needed for launch.

Supabase, Cloud Run & Firebase

~1 day

9.1 Supabase project

  • Create project in eu-north-1 (Stockholm) — closest Supabase region to Tallinn.
  • Create the public place-photos Storage bucket; generate S3 access keys (Storage → Settings).
  • Grab two connection strings from Connect: the transaction pooler (:6543) for the app, and the session pooler / direct (:5432) for migrations.
Pooler vs direct — the #1 footgun Cloud Run scales horizontally; each instance opens a Hikari pool. Route the app through the transaction-mode pooler (6543) and keep spring.datasource.hikari.maximum-pool-size small (≈5). Transaction mode does not support server-side prepared statements, so add ?prepareThreshold=0 (and disable Hibernate statement caching) or you'll get errors under load.
Run Liquibase on the direct/session connection (5432), not the transaction pooler — DDL + advisory locks need a session. Point spring.liquibase.url at 5432 and spring.datasource.url at 6543.

API contract verification checklist

The acceptance gate. The Spring API must be indistinguishable from the mock/Nest to the SPA. Verify against the deployed Cloud Run URL.

  • GET /health → {"status":"ok"}, 200.
  • GET /categories → 7 items, order tabletennis→sakura, each {id,label,short,color,glyph} with color = cssvar string, glyph = id.
  • GET /places → 10 ACTIVE items, ids "01".."10" in sort order; each row has lat/lng, category object, tags[], no rating field.
  • GET /places/{slug} → detail with note, photos[], viewsCount, sharesCount, contributor.name, verifiedAt (ISO or absent), fieldNotes{access,lit,best}, appleMapsUrl, googleMapsUrl.
  • GET /guides → "Free to play" first, then per-category guides with ≥2 spots; getGuide resolves spots.
  • POST /events → 202.
  • Auth round-trip: register → 201 with tokens; login → 200; /auth/me with bearer → user; refresh rotates; bad refresh → 401.
  • Auth gating: unauth POST /submissions|/reports|/saved → 401.
  • Saved cross-device: save on device A, GET /saved on device B (same login) returns it; merge unions guest ids.
  • Submission lifecycle: POST /submissions → PENDING → admin approve → appears in public GET /places as ACTIVE with next padded id.
  • Upload: POST /uploads (image) → {url} on Supabase; URL persists after a Cloud Run redeploy.
  • Admin gating: non-admin token on /admin/** → 403; admin token → data.
  • End-to-end: point a local web build at the Cloud Run URL — Explore, detail, guides, save, submit, login all work with zero frontend code changes.

Cutover & rollback

  1. Build the Spring backend in the monorepo; ./gradlew test green.
  2. Provision Supabase (DB + bucket); run Liquibase against :5432; seed.
  3. Deploy the container to Cloud Run; verify the checklist against the run.app URL directly.
  4. Deploy the web build to Firebase with VITE_API_URL = Cloud Run URL; set CORS_ORIGIN to the web.app origin.
  5. Re-run the end-to-end checklist against the live web.app site.
  6. Retire Nest: delete the Render web service + Render Postgres, the old Pages workflow, and backend/ Node files + render.yaml. Tag the cutover commit (e.g. v0.3.0).
Real data on Render?The seed data is static, so the only data worth keeping is any production users / saves / submissions created on the live Render DB. If there is any, pg_dump the Render Postgres and restore the relevant tables into Supabase before deleting Render. If Render only ever held seed data, skip this — the Supabase seed reproduces it.

Rollback: until you delete Render, rollback is a one-line revert of the VITE_API_URL secret + a Pages redeploy — the old stack is still live. After deletion, rollback means redeploying the tagged Nest commit. Keep Render running for a short bake period before pulling it.
