GemSpot UI Specification

A developer-ready, screen-by-screen implementation spec for the GemSpot React MVP. It does not redesign anything — it converts the approved Direction C concept and the design system into concrete layout, component, state, and responsive rules a frontend engineer can build directly against.

5 flows · browse · filter · detail · save · add 900px breakpoint Mobile-first · desktop split-view React + TS · MapLibre / Mapbox
Reading this as: a map-first local-discovery web app for technical handoff. The map is the stage; chrome stays calm; one accent blue carries every action; category colour lives only on markers, chips, and badges. Every list, detail, and map ships loading / empty / error states — those are the product for a map + UGC app, not polish.
01 — Orientation

Product UI overview

GemSpot is one persistent app shell, not a set of separate pages. The map never unmounts; screens are states layered over it.

  • One shell, layered states. AppHeader + a route outlet + (mobile) MobileTabBar + global AuthDialog & Toaster. Explore, Saved, and Spot detail all share the same map instance — detail is an overlay/panel, never a fresh page that unmounts the map.
  • Desktop = split view. A persistent left rail (416px) of result cards beside a map that fills the rest. Spot detail slides over the rail; the map stays put and recenters on the selected pin.
  • Mobile = map + bottom sheet. Full-bleed map, a draggable bottom sheet for results & previews, a floating filter chip row up top, and a 4-item bottom tab bar (Explore · Saved · + · Profile) with the centre Add FAB.
  • Public to view, account to act. Browse / filter / open spots / per-spot pages are public and SEO-indexable. Login gates only Save, Add, /saved, /admin. Gated actions queue and resume after login (requireAuth).
  • UGC is moderated. Submissions go to a PENDING queue and are reviewed in a separate, utilitarian desktop admin surface before they ever appear on the map. (Admin is out of scope for this consumer spec — see the design system inventory.)
!
13 UX decisions are locked (docs/07) and constrain this spec — don't silently revert them: bookmark is the only save metaphor (no hearts); category chips are the only filter; no ratings/stars in MVP; distance is shown and is the default sort; Directions hands off to Apple/Google Maps; Tallinn only at launch.

11 — Behavior

Interaction rules

14 — Handoff

Missing assets & open design questions

What must be resolved or produced before (or alongside) frontend implementation. None block starting the shell, but each blocks a specific surface from being "done".

Real map style

The hand-drawn SVG map is a stand-in. Need a MapLibre/Mapbox custom style matching the palette (land #eef1f6, water #cfe0f2, parks #d3e7ca, muted labels) + the 4 marker states as real map symbols.

Design + FE · blocks Explore

Photography

No real spot photos exist yet — every card/detail uses the category-tint PlacePhoto fallback. Need a sourcing pipeline + aspect ratios (hero 16:10, thumb 1:1) and an OG-image rule for SEO.

Content · blocks Detail polish

Launch category subset

Prototype ships 7 categories; brief says 3–5. Pick the launch set (and their glyphs/hues) — keep data-driven so the rest are config, not code.

Product · blocks Filter

Full state art

Loading skeletons (rail rows, detail hero), search no-match, tiles-failed, out-of-region, report-sent — listed but not all drawn. For a map+UGC app these are scope.

Design · blocks state-complete

Auth screens

AuthDialog content (social buttons, email form, error copy) and the deep-link page variants (/login, /register) need final layout + provider list confirmation.

Design + FE · blocks Save/Add

Report flow

"Report / no longer here" entry point, form fields, and the report-sent state are referenced but not fully specified.

Product + Design

Search

The top-bar search field exists visually; query behavior, results UI, and the no-match state need definition (or scope it out of v1 explicitly).

Product · decide scope

Icon set decision

Keep the curated custom stroke set or adopt one library (e.g. Phosphor) at a single stroke weight. Category glyphs stay as brand assets either way.

15 — Handoff

React component breakdown

FSD-ish layers. Layout wrappers carry the responsive swap so feature components stay breakpoint-agnostic. State: thin global store (Zustand/Jotai) for cross-cutting UI; TanStack Query for server data; URL is the source of truth for route, filter, and selected spot.

// app shell — the responsive frame everything mounts into <AppShell> <AppHeader/> // desktop top bar; collapses <900 <RouteOutlet> <ExploreLayout> // ← the responsive wrapper: rail|sheet swap <CategoryFilter value=url onChange=url /> <ResultsContainer> // renders into rail (desktop) or BottomSheet (mobile) <SpotListItem .../> // × n (entities/place) </ResultsContainer> <SpotMap places selectedId onSelect pinMode> <SpotMarker/> <ClusterMarker/> </SpotMap> <SpotDetail variant="panel|page" /> // /spot/:slug </ExploreLayout> <SavedPage/> // reuses ExploreLayout, saved-only set <AddSpotFlow/> // 4-step; FlowLayout (panel|stacked) </RouteOutlet> <MobileTabBar/> // <900 only; Add = centre FAB <AuthDialog/> <Toaster/> // global </AppShell>

Responsive layout wrappers (build these first)

  • AppShell — owns the breakpoint; decides header vs tab-bar, dvh handling.
  • ExploreLayout — rail+map (desktop) ↔ map+sheet (mobile); hosts ResultsContainer + SpotMap + SpotDetail.
  • ResultsContainer / BottomSheet — the sheet↔rail swap; one content tree, two shells.
  • FlowLayout — Add's panel-beside-map (desktop) ↔ map-top + panel (mobile).

State homes

  • URL — route, ?cat=, selected /spot/:slug.
  • Auth store — user, role, requireAuth + queued action.
  • TanStack Query — places, categories, favorites (optimistic).
  • UI store — sheet snap, toast, location permission.
16 — Handoff

Recommended implementation order

Build the shell and the map first — they de-risk everything downstream. Effort tags: easy medium hard.

17 — Handoff

Platform-specific edge cases

Decisions that need an explicit answer for one platform before that surface ships.

Mobile-specific mobile
  • Sheet vs keyboard: when a field in a sheet/flow focuses, the on-screen keyboard must not bury the primary CTA — push the CTA above the fold or float it.
  • Map height in Add: at ~38% map height the pin tap-target gets small; guarantee ≥44px and allow pinch-zoom before placing.
  • dvh / browser chrome: bottom tab bar must clear the iOS Safari address bar — 100dvh, test on device.
  • Sheet over FAB: at full snap the sheet must not occlude the tab bar / Add FAB, or define how they coexist.
  • Native share: Share uses the OS share sheet; Directions uses app deep links (Apple Maps default on iOS, Google on Android?) — confirm.
Desktop-specific desktop
  • Panel + map width: when the detail panel is open over a 416px rail, define the minimum map width before the panel goes full-rail-width (small laptops).
  • Hover vs touch: marker hover/preview is desktop-only — gate behind (hover:hover) so hybrid touchscreens don't fire false hovers.
  • Deep-link cold load: /spot/:slug opened cold on desktop — does it render the panel over a default Explore, or a standalone page? (Spec says panel-over-explore.)
  • Keyboard map nav: pan/zoom + marker focus order for keyboard-only users; Esc closes panel before leaving the map.
  • Very wide screens: cap the map content / rail max-width so the layout doesn't stretch awkwardly past ~1800px.
