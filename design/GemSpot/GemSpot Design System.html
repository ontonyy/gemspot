GemSpot Design System

A small, production-ready system extracted from the existing Direction C concept — a clean geo-product with playful accents. Nothing here is redesigned: the visual language, tokens, and components are taken verbatim from the prototype and normalized into reusable, React-ready rules scoped to MVP.

One accent · #2f6fe0 Type · Space Grotesk + Hanken Grotesk 7 category hues Map-first · mobile + desktop Light + dark mode
01 — Foundations

Design principles

Five rules decide every component below. When a choice is ambiguous, these break the tie.

  • The map is the stage, not the wallpaper. Chrome stays calm and neutral so markers, cards, and the accent carry all the signal. Surfaces never compete with the map.
  • One accent, decorative colour is data. --accent blue is the only action colour — buttons, links, selection, active nav. Category colours appear on markers, chips, and badges only; a category hue never becomes a CTA.
  • Every category reads twice — colour and glyph. Markers and chips pair a hue with a distinct icon shape so they survive colour-blindness, greyscale, and tiny sizes.
  • States are the product, not polish. For a map + UGC app, loading / empty / denied / pending / error are core scope. Each component ships its full state set.
  • Motion preserves spatial continuity. Selecting a pin, opening a panel, expanding a sheet — motion exists to keep the user oriented, stays under 300ms, and respects reduced-motion. It is never decoration.
!
13 UX decisions are locked (see docs/07) and shape this system: bookmark is the single save metaphor (no hearts), category chips are the only filter, no ratings/stars in MVP, submissions go to a moderation queue, and Tallinn is the only city at launch. Don’t silently revert these in the build.
02 — Foundations

Color system

The entire UI runs on one neutral ramp plus one accent. Cool, low-chroma greys keep the map legible; the accent does all the work of signalling action and selection.

Neutrals & accent

Semantic status — add as tokens (used in prototype, not yet in :root)

Category palette — markers & badges only

Seven curated hues, each locked to a glyph. They identify content; they are never used for buttons, links, or backgrounds. MVP ships 3–5 of these (launch subset pending); the system supports any subset because categories are data-driven.

i
Each category is recognisable by both colour and glyph. Keep that rule when adding a category — pick a hue that is distinct in greyscale and a glyph shape that is not already in use.

Dark mode

The system is dark-ready: every surface reads from the same role tokens, so dark mode is a ramp swap, not a re-skin. The neutral family goes cool-dark, the accent brightens to stay recognisable and hold WCAG AA on dark surfaces, and category hues are unchanged — the white marker ring carries their contrast over any background. Flip the toggle in the top bar to preview the whole page.

i
Don’t ship a half-built dark mode — these values are the baseline. Audit marker, chip, and CTA contrast in dark before release (it’s folded into the a11y pass in Design next).

13 — Map & navigation

Map-specific UI

The marker is the most brand-defining element in the product. Every marker is a filled circle in its category hue, a 2.5px white ring, a soft tinted shadow, and a centred white glyph (~62% of the marker). The white ring is what guarantees contrast over any map surface.

Pin states

Selected pin behaviour

On select the marker scales to ~1.32×, gains a translucent colour halo and a white name-label, and the map gently recenters toward it — reinforcing “this pin = this card”. Only one marker is selected at a time.

Clusters & map controls

!
Clustering is required, not optional. At city zoom, nearby pins collapse into a count pill (up to 3 colour dots + count). In production a cluster click zooms to bounds; the popover list is only the max-zoom fallback. Without this, Old Town scenic pins pile into an unreadable stack.
!
Normalize the saved pip: the prototype marker still uses a heart pip — the locked metaphor is bookmark everywhere. The pin above shows the corrected bookmark pip; carry that into production.

Sheet behaviour

  • Two snap heights — half (46%) for preview, full (88%) for the list. A grab handle signals draggability.
  • Drag-interactive in production — spring + velocity dismissal (v > 0.11), not just a tap-to-toggle CSS height change. Use motion values, not per-frame state.
  • The map never disappears behind the sheet; selecting a row in the sheet still drives the pin.
  • Sheets are for contextual, short content — place preview & results. Long flows (Add a spot) get a dedicated screen, not a sheet.

Navigation set

  • Explore · Saved · Add · Profile — four destinations, no more. Add is the centre FAB on mobile.
  • Active = accent; inactive = ink-faint. Saved shows a count dot when populated.
  • Admin is not in this nav — it’s a separate, role-gated desktop surface.

Dark theme override

Same token names, swapped values. Apply on a [data-theme="dark"] attribute (or under @media (prefers-color-scheme: dark)). Components don’t change — they already read the roles.

/* dark ramp — neutrals swap, accent brightens for AA */
[data-theme="dark"] {
  --bg: #0d1219;      --surface: #151c27;   --surface-2: #1e2735;
  --ink: #eaf0f8;     --ink-soft: #9fabbb;  --ink-faint: #66717f;
  --line: #2b3543;    --accent: #4d8bf2;   --accent-2: #29b3b3;
  color-scheme: dark;
  /* status + radius + type + category hues: unchanged */
}

18 — For engineering

Suggested token naming for React

Semantic, role-based names — not raw colour names. Components reference roles (surface, ink-soft, accent), so a future dark mode or rebrand swaps values without touching component code. Two-level dot/dash scale, kebab-case CSS vars mirrored by a typed TS object.

// tokens.ts — typed mirror of tokens.css
export const token = {
  color: {
    bg: 'var(--bg)',
    surface: 'var(--surface)',
    surfaceMuted: 'var(--surface-2)',
    ink: 'var(--ink)',
    inkSoft: 'var(--ink-soft)',
    inkFaint: 'var(--ink-faint)',
    line: 'var(--line)',
    accent: 'var(--accent)',
    status: { success, warning, danger },   // semantic, not raw
  },
  radius: { tag, chip, btn, card, sheet, pill },
  space: { 1: 4, 2: 8, 3: 12, 4: 16, 5: 24, 6: 32 },
  font: { head, body },
  ease: { out, drawer, marker },
} as const;

// category hue is DATA, kept out of the theme:
export const categoryColor: Record<CategoryId, string> = {
  tabletennis: '#e0a32e', basketball: '#ee7733', football: '#2f9e5b',
  tennis: '#74b13a', padel: '#1ba0a0', scenic: '#3b7ddd', sakura: '#e85f9c',
};
  • Roles over hues. A component asks for ink-soft, never #5c6675. Theme owns the mapping.
  • Category colour is data, not theme. It comes from the API (GET /categories), keyed by id — never a hard-coded button colour.
  • One scale, two formats. kebab-case CSS variables for styles; a typed token object for TS/props. They mirror 1:1.
  • Component variants are enums — <Button variant="primary|secondary|ghost">, <Sheet snap="half|full">, <Marker state="default|selected">.
19 — For engineering

What to design next

Three things will define usability and uniqueness most, and aren’t fully resolved yet. Tackle in this order.

1

Real map style & live markers

The hand-drawn SVG map is a stand-in. Build the MapLibre/Mapbox custom style matching the palette, and render the four marker states as real map symbols — including cluster zoom-to-bounds. This is the make-or-break of the product.

2

The full state set, drawn

Loading skeletons (rail rows, detail hero), empty (filter / saved / search), location denied & out-of-region, submit→pending, report-sent, tiles-failed. For a map + UGC app these are the product, not edge cases.

3

Accessibility & dark-mode polish

The dark ramp now ships in tokens — next is the audit: marker / chip / CTA contrast over every map surface in both themes, full keyboard + visible-focus coverage, ≥44px targets, and reduced-motion fallbacks. Hold WCAG AA throughout.

i
Lower priority but reserved in schema: collections / curated guides, the advanced filter panel, and ratings — each is a deliberate scope expansion to discuss, not a default add.
